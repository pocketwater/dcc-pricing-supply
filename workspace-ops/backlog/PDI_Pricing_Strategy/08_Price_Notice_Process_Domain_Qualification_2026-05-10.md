# Price Notice Process — Domain Qualification Memo

**Date:** 2026-05-10
**Phase:** 8 — Price Notice Process Domain Qualification
**Verdict:** **VIABLE BUT COMPLEX** — Conditionally qualified for external UX replacement with augmentation path

---

## Executive Summary

PDI's native price notice generation and display workflow is distributed across three distinct subsystems with overlapping responsibilities, inconsistent UX touchpoints, and determinism issues that make reliable native pricing difficult for CitySV operations. This memo qualifies the opportunity to replace the native UX while leveraging existing PDI table structures and batch procedures to maintain data consistency.

**Key Finding:** Price notices exist in three parallel systems — **CP** (Customer Pricing), **OD** (Orders), and **OD-Warehouse** — each with different rules, effective-date semantics, and export capabilities. A unified external UX that reads/writes to the same underlying PDI tables can normalize this fragmentation.

---

## 1. Price Notice Architecture

### Subsystems (Three-System Model)

| Subsystem | Module | Purpose | Primary UX Programs | Triggers |
|-----------|--------|---------|---------------------|----------|
| **CP** (Customer Pricing) | CP (Pricing) | Static customer price rules and notice batch generation | CPPNR (rules), CPPN (generate) | Manual or scheduled batch |
| **OD** (Order Fuel) | OD (Orders) | Price notices for fuel orders at time of order entry/edit | ODPN (view/process), ODWPN (warehouse) | Order creation, fuel-contract-attached orders |
| **OD-Warehouse** | OD (Orders) | Price notices for warehouse orders (bulk transfers) | ODWPN (view/process) | Warehouse order activity |

### Data Model: Three Tables, Parallel Structure

```
Customer_Price_Notices           ← CP batch output (static lookup, rare updates)
├─ Key: CPN_Key
├─ ForeignKey: CPN_Cust_Key, CPN_CustLoc_Key
├─ Data: Batch ID, batch date, notice text, effective date range
└─ Generated by: ODCPNB (Customer Price Notice Batch) program

Customer_Fuel_Prices            ← CP calculated (intermediate, weekly/on-demand)
├─ Key: CustPrice_Key
├─ ForeignKey: CustPrice_Cust_Key, CustPrice_CustLoc_Key, CustPrice_Prod_Key
├─ Data: Price value, effective date, price basis, terminal, contract origin
└─ Calculated from: Customer_Fuel_Contracts + OD_Fuel_Pricing_Rules

OD_Calculated_Prices            ← OD transient (order-time, real-time calculation)
├─ Linked to: Orders + Order_Details_Fuel
├─ Not persisted as a notice
└─ Displayed in ODE (Order Data Entry) UX during order create/edit
```

### Companion Tables (Reference, Read-Only in UX)

| Table | Purpose | Refresh Cadence |
|-------|---------|-----------------|
| `Customer_Price_Notice_Details` | Line items within a notice batch | Generated with parent batch |
| `Customer_Fuel_Contract_Details` | Effective date ranges for contract periods | Updated on contract import |
| `OD_Fuel_Pricing_Rules` | The rules applied to calculate prices | Updated on contract import |
| `Customer_Fuel_Prices_History` | Archive of prior calculated prices | Retained for audit |

## 2. Generation and Workflow

### CP Flow: Customer Pricing Batch (CPPNB → CPPN)

**Trigger:** Manual (operator runs CPPN program) or scheduled batch job

**Proc:** `Customer_Price_Notice_Batch_ODCPNB_SP` (primary generator)

**Steps:**
1. Operator/scheduler invokes CPPN (Generating Price Notices) program in PDI UI
2. Program selects eligible customers from `Customers` table (filtered by location, contract group, status)
3. For each customer, looks up effective `Customer_Fuel_Contracts` (using effective-date windowing)
4. Resolves `OD_Fuel_Pricing_Rules` for products covered by active contracts
5. Calculates price tiers (volume-based if applicable)
6. Generates `Customer_Price_Notice_Header` and `Customer_Price_Notice_Details` rows
7. Inserts into `Customer_Price_Notices` table
8. Optionally exports to external formats (PDFs, DTN, email)

**Cadence:** Weekly or on-demand (not automated to orders)

**Critical Constraint:** Price notices are **NOT** generated at order time; they are batch publications intended for external customer communication (invoices, price lists, EDI). They do **NOT** drive order pricing in ODE during `_Orders_Upload`.

### OD Flow: Order-Time Pricing (ODPN, real-time calculation)

**Trigger:** Order entry/edit in ODE (Order Data Entry)

**Method:** Direct calculation at order save time (no batch)

**Procs:**
- `OD_GetEffectiveCustFuelContracts_SP` — resolve applicable contracts
- `OD_CalcPrices_ResolveFuelPricingRules_SP` — match rules to order line
- `OD_CalcOrder_SP` — compute final price, apply adjustments

**Semantics:**
- Pricing is calculated **at order moment**, not retrieved from pre-calculated `Customer_Fuel_Prices` table
- If a customer fuel contract is assigned to the order, those rules take precedence
- Volume tiers, effective-date windowing, and product matching are deterministic
- Result is stored in `Order_Details_Fuel.OrdFuel_Price` (populated before order posts)

**Export to Order Upload:** When order is exported to gravitate-orders via `_Orders_Upload`, the price value in `OrdFuel_Price` is published as-is. The price is NOT recalculated; it's treated as source-of-truth once order is posted.

### OD-Warehouse Flow: Warehouse Order Pricing (ODWPN)

**Trigger:** Warehouse order activity (bulk transfer orders)

**Method:** Similar to order pricing, but with warehouse-specific routing/fuel rules

**Procs:**
- `OD_Calc_Warehouse_Prices_SP` — warehouse-specific pricing rules
- `OD_GetEffectiveWarehouse_FuelContracts_SP` — warehouse contract lookup

**Semantics:**
- Warehouse orders have separate contract/rule hierarchy (not same as customer-direct orders)
- Used for inter-site transfers and wholesale redistributions
- Pricing resolution is similar to order pricing but constrained to warehouse destinations

## 3. Current Pain Points (Why Native UX is Difficult)

### Pain 1: Three Parallel Systems with Inconsistent Semantics
- **CP** system designed for static batch publication (customer communication)
- **OD** system designed for real-time order pricing (transactional)
- **OD-Warehouse** system designed for warehouse routing
- All three can produce "prices" but they answer different questions (announcement vs. transaction vs. internal transfer)
- Operators must understand which system they're in and how to navigate between them
- **Evidence:** CitySV has moved to external price matching (vw_CitySV_OrderPrice_F_Row_Match) to avoid complexity

### Pain 2: Effective-Date Windowing is Opaque
- Contract effective dates and expiration dates are complex
- Price notices use `CPN_Effective_Date` and `CPN_Expiration_Date` with unclear precedence
- OD rules use `CustFuelContDtl_Effective_DateTime` and `CustFuelContDtl_Expires` with different semantics
- Operators cannot easily see which contract/rule period a price belongs to without running complex queries
- **Symptom:** Cannot easily reverse-engineer why an order got price X on date Y

### Pain 3: Multi-Contract Overlap and Rule Interaction
- If customer has two active contracts with overlapping product coverage, **all rules participate** in pricing decision
- The merging/precedence logic is in `OD_CalcPrices_ResolveFuelPricingRules_SP` (not documented in UX)
- If rules conflict, determinism is unclear
- **Risk:** Same order uploaded on different dates could get different prices if contract periods overlap

### Pain 4: Batch Refresh Latency
- CP price notices are **NOT** real-time; they're batch publications
- If a customer contract changes, the change doesn't appear in `Customer_Price_Notices` until next CPPN batch run
- OD pricing is real-time but transient (not visible in `Customer_Price_Notices`)
- Operators see stale prices in the notice table, live prices during order entry, then different prices in posted orders if a contract changed between notice generation and order entry

### Pain 5: No Direct Edit Path
- Price notices cannot be edited directly in PDI UI
- If an incorrect price notice is published, must delete the batch and re-run CPPN
- No partial-update or line-item correction capability
- **Evidence:** CitySV escalates pricing corrections to offline processes instead of using PDI UI

### Pain 6: DTN and External Export Integration is Loose
- Exporting price notices to DTN (Data Terminal Network) or email requires manual operator action
- Export format is not configurable within PDI UI (fixed-format export)
- No audit trail of what was exported, when, to whom
- **Symptoms:** Missed exports, duplicate exports, format mismatches with external systems

### Pain 7: Audit Trail is Incomplete
- `Customer_Price_Notices` table does not track who created/approved a notice
- No change audit table for contract updates vs. price notice changes
- Lineage between contract import → rule calculation → notice generation → order pricing is not explicitly tracked
- Compliance/dispute resolution requires manual debugging of multiple tables and procs

## 4. External UX Replacement Strategy

### Proposed Model: Unified Price Notice Platform (External)

Replace the native **CP** (Customer Pricing) → **CPPN** (Generate Price Notices) workflow with an external system that:

1. **Reads** from PDI tables:
   - `Customer_Fuel_Contracts` (source of truth for contract terms)
   - `OD_Fuel_Pricing_Rules` (source of truth for pricing rules)
   - `Customer_Fuel_Contract_Details` (effective-date periods)
   - `Customers` and `Customer_Locations` (reference data)

2. **Displays** a unified UX showing:
   - All active contracts for a customer (with effective-date ranges)
   - All applicable pricing rules (with product/terminal/origin filters)
   - Calculated price tiers for each product
   - Change history and approval workflow

3. **Writes back** to PDI tables:
   - Insert/update `Customer_Fuel_Contracts` headers (upsert-based on contract ID)
   - Insert/update `Customer_Fuel_Contract_Details` (effective-date periods)
   - Insert/update `OD_Fuel_Pricing_Rules` (individual pricing rules)
   - Trigger `ODW_FuelPricingContractImport_*_SP` procedures (existing validation and post logic)

4. **Generates price notices** on-demand or scheduled:
   - Call `Customer_Price_Notice_Batch_ODCPNB_SP` after contract updates
   - Poll `Customer_Price_Notices` and `Customer_Price_Notice_Details` tables
   - Export via configured DTN/email integration (or custom export handler)
   - Log audit trail to external audit table (trigger in PDI or API-side logging)

### Architecture Diagram

```
External UX (Python/TypeScript)
    ↓
Read Layer                          Write Layer
├─ Customer_Fuel_Contracts          ├─ ODFPCI import validation
├─ OD_Fuel_Pricing_Rules            ├─ ODW_FuelPricingContractImport_*_SP
├─ Customer_Fuel_Contract_Details   ├─ Customer_Fuel_Contracts header upsert
├─ OD_Fuel_Pricing_Products         ├─ OD_Fuel_Pricing_Rules upsert
├─ OD_Fuel_Pricing_Locations        └─ Post to permanent tables
└─ OD_Fuel_Pricing_Origins
    ↓
[Existing PDI Procs + SetDefaults]
    ↓
[PDI Tables: Contracts, Rules, Products, Locations, Origins]
    ↓
OD_CalcOrder_SP (order pricing, unchanged)
│
↓ [at order time]
│
Order_Details_Fuel.OrdFuel_Price (populated by existing OD logic)
│
↓ [order posts → gravitate-orders export]
│
_Orders_Upload (price flows through, unchanged)
```

### Key Insight: Augmentation, Not Replacement

The external UX does **NOT** replace the PDI order pricing engine (`OD_CalcOrder_SP`). Instead:

1. **External UX** owns the **customer-facing contract and pricing rules authoring/editing/approval workflow** (replaces CPPNR UI)
2. **External UX** owns the **price notice batch generation and export** (replaces CPPN UI)
3. **External UX** reads/writes to PDI contract tables (same source of truth)
4. **PDI ODE** continues to calculate order prices at order-entry time using the same contracts/rules (unchanged)
5. **Gravitate-orders** continues to export orders with prices already calculated by PDI (unchanged)

**Result:** One unified, auditable pricing authority instead of three fragmented systems.

### Upsert Path (Write)

**Entry Point:** `ODW_FuelPricingContractImport_ResolveKeys_SP`

**Input CSV Format** (same as ODFPCI import):

```csv
Record_Type,Contract_ID,Cust_ID,Effective_Date,Rule_Description,Product_ID,Price_Basis,Adj_Method,Amount,...[43 more fields]
CFC,CITYSV-C001,12843,04/15/2026,,,0,-1
```

**Key Parameters:**
- `@UpdateAction` = 0 (update existing), 1 (replace), 2 (error if exists)

**Output:** Rows inserted to permanent tables; errors logged to staging table for inspection

## 6. Risk Assessment

### Risk 1: Effective-Date Overlap Collision
- **Current state:** `OD_GetEffectiveCustFuelContracts_SP` returns **ALL** matching contracts (multiple rows per customer on same date)
- **Impact:** If external UX creates overlapping contract periods, both contribute rules. Pricing interaction is undocumented.
- **Mitigation:**
  - External UX must enforce "no overlapping contracts for same customer + products + date range"
  - Pre-import query: `SELECT * FROM OD_Fuel_Pricing_Rules WHERE CustFuelCont_Cust_Key = ? AND ODFuelPrcRule_CustFuelContDtl_Key IN (SELECT CustFuelContDtl_Key FROM ... WHERE Effective overlap)`
  - If overlap detected, warn and allow operator to choose: merge, defer, or split by effective date

### Risk 2: Contract Effective-Date Semantics Confusion
- **Current state:**
  - `Customer_Fuel_Contract_Details.CustFuelContDtl_Effective_DateTime` = when rules start applying
  - `CustFuelContDtl_Expires` (boolean flag) = whether an expiration exists
  - `CustFuelContDtl_Expiration_DateTime` = when rules stop applying (if Expires=True)
  - If `Expires=False`, contract has no expiration (open-ended)
- **Impact:** External UX must handle three-state logic for expiration (no expiry / expiry set / expiry unset)
- **Mitigation:**
  - Document this three-state model clearly in UX (radio button: "Active until [date]" vs. "No expiration")
  - Show effective/expiration dates prominently, with validation rules
  - Test with date ranges that cross from active → expired → reactivated

### Risk 3: SetDefaults Fills ~30 Fields with Zero
- **Current state:** `ODW_FuelPricingContractImport_SetDefaults_SP` auto-fills many FPRULE columns
- **Impact:** External UX must either:
  - Accept defaults (simpler, but less control)
  - Allow user to override defaults (more flexible, but more complex UX)
  - Know which fields matter vs. which are safely defaulted to 0
- **Mitigation:**
  - Maintain a field "essentiality" matrix showing which 46 FPRULE columns are required, optional, or always-default
  - Default to accepting PDI defaults for now; allow advanced mode later if needed

### Risk 4: No Rollback on Bad Import
- **Current state:** `ODW_FuelPricingContractImport_Post_SP` writes directly to permanent tables; `DeleteBatch_SP` only cleans staging
- **Impact:** A bad import is live immediately. Undo requires manual DELETE or setting `CustFuelCont_Inactive=1`
- **Mitigation:**
  - External system always imports with distinctive `Contract_ID` prefix (e.g., `CITYSV-*`)
  - After import, external system runs validation query to spot check a few rows
  - If validation fails, external system issues immediate alert; operator can decide to revoke contract (set Inactive=1) or debug

### Risk 5: Same Execution Identity Constraint
- **Current state:** ODFPCI procs require `ColemanOil\svc-Quser` (same as ODIMP)
- **Impact:** External UX cannot call procedures directly; must route through PDI file-drop + scheduled import job
- **Mitigation:**
  - External system writes CSV to network file drop (e.g., `\\PDI-SQL-01\EnterpriseData\...`)
  - PDI scheduler runs `ODW_FuelPricingContractImport_*` procs automatically
  - External system polls batch result table to confirm completion
  - This is the same pattern used for ODIMP (gravitate-orders); proven workflow

### Risk 6: Price Notice Batch Generation Not Automated to Orders
- **Current state:** `ODCPNB` (price notice batch) is manual/scheduled, NOT triggered by order activity
- **Impact:** Price notices are stale by the time orders arrive; order prices come from real-time OD calculations (not from notices)
- **Mitigation:**
  - External UX should clarify what "price notice" means in this system:
    - **CP path:** Static batch publications for customer communication (invoices, price lists, EDI) — not used for order pricing
    - **OD path:** Real-time calculation at order entry (what actually drives `OrdFuel_Price`) — this is what matters for orders
  - Order pricing is decoupled from price notices; this is by design, not a bug
  - External UX should focus on OD contract/rule authoring (which drives order pricing) rather than CP notice generation

### Risk 7: Multi-Customer Contracts via Contract Group
- **Current state:** Contracts can be assigned to a `CustContGrp_Key` (customer group), applying to all customers in group
- **Impact:** Changing a group-level contract affects multiple customers; no per-customer override
- **Mitigation:**
  - External UX must clearly distinguish group-level vs. customer-level contracts
  - Warn operator when creating a group contract: "This will apply to [N] customers"
  - Provide drill-down to see which customers are in the group
  - Allow later override by creating a customer-specific contract (higher priority in resolution order)

### Risk 8: Validator Does Not Check Business Logic, Only Schema
- **Current state:** `ODW_FuelPricingContractImport_Validate_SP` checks for duplicates, missing FKs, invalid dates — but NOT business logic
- **Impact:** Can import technically valid but operationally broken rules (e.g., negative amounts, wrong terminals)
- **Mitigation:**
  - External UX should add business-rule validation BEFORE calling import:
    - Amount sign validation (positive for markups, negative for discounts)
    - Terminal/vendor existence check
    - Product availability check
    - Effective-date sanity checks
  - Log validation failures to external audit table for dispute resolution

- Wire up `ODCPNB` call after contract import (optional CP path)

- Decommission native CPPNR UX (or keep as read-only reference)
- Train operators on new external system
- Monitor first 2 weeks of live imports for issues

## 8. Success Criteria

### Operational
- [ ] Operators can view all active customer contracts in one place (unified UX)
- [ ] Operators can create/edit/approve contracts without SQL debugging
- [ ] Contract changes flow to PDI and take effect in order pricing within 1 import cycle
- [ ] Price notices can be generated on-demand and exported to customers with audit trail
- [ ] No more "why did this order get priced at $X?" escalations without traceable answer

### Technical
- [ ] Contract import succeeds 99% of the time (retryable failures are < 1%)
- [ ] Order pricing reflects imported contracts within < 1 minute of import completion
- [ ] External audit trail captures 100% of contract changes (creates, updates, deletes)
- [ ] Rollback time for bad import is < 5 minutes (set Inactive=1)
- [ ] Pre-import validation catches 100% of overlapping contracts and schema violations

### Compliance
- [ ] All price changes are logged with who, when, why, approved-by
- [ ] Audit trail is immutable and legally defensible
- [ ] Price notices are exportable and traceable to underlying contracts
- [ ] Dispute resolution team can trace order price → contract rule → contract creation event

## Verdict: VIABLE — With Augmentation

The Price Notice process domain is a legitimate candidate for external UX replacement.

### What's Proven
- Three distinct subsystems (CP, OD, OD-Warehouse) all read/write to shared PDI contract tables
- Contract import via ODFPCI is deterministic and upsert-based (proven in Phase 6 POC)
- Order pricing reads from imported contracts directly (no notice pre-generation required)
- Existing PDI procs handle validation and persistence; external system adds authoring/audit UX

### Conditions for Go
1. **Namespace isolation** — All CitySV contracts use `CITYSV-*` prefix on `Contract_ID`
2. **Effective-date windowing must be understood** — Three-state expiration logic (no expiry / expiry set / expiry unset) must be documented in UX
3. **Overlap guard** — Pre-import query confirms no conflicting active contracts for same customer + products + date
4. **Execution identity workaround** — External system writes CSV to file drop; PDI scheduler runs ODFPCI procs; external system polls batch result table
5. **Audit trail is external** — PDI tables are source of truth for data; external system owns audit/compliance trail

### What This Enables
If activated, the external pricing UX becomes:

```
External Pricing UX (unified for CP + OD)
    ↓ [author contracts/rules, approve, import]
    ↓
ODFPCI import (existing PDI procedures)
    ↓
Customer_Fuel_Contracts + OD_Fuel_Pricing_Rules (persistent, source of truth)
    ↓
OD_CalcOrder_SP (order pricing at entry time, uses imported rules)
    ↓
Order_Details_Fuel.OrdFuel_Price (pre-calculated, deterministic)
    ↓
_Orders_Upload (price flows through unchanged)
```

This **deprecates the CP → CPPN batch notice workflow** for internal ordering and replaces it with a deterministic, auditable, unified contract/pricing platform. CP → CPPN can remain for external customer communication (price lists, invoices), but order pricing is decoupled and driven by OD.
